BMO build notes

1.8" ST7735 TFT face on ESP32-C3 Super Mini, with I2S audio, touch reactions, and pre-baked voice clips.

Full wiring

Everything plugs into one ESP32-C3 Super Mini. Power rails (3V3, 5V, GND) are shared via breadboard rails. Signal pins are dedicated.

I2S sharing — why mic and amp share clock pins

The C3's I2S peripheral is half-duplex: it can record OR play, not both at once. The clock signals (BCLK and LRC) are identical for input and output, so we wire both the mic and amp to the same GP0/GP1. Only the data pins are separate (GP2 for amp out, GP5 for mic in). Software switches between record and play modes.

Troubleshooting

Screen shows white / black / nothing

  1. Run the serial monitor first to confirm firmware is running: 
    pio device monitor -p /dev/cu.usbmodem134301 -b 115200
    You should see mood: idle, mood: blink, etc. scrolling.
  2. Power-only test: disconnect every signal wire (CS, RESET, A0, SD, SCK), leave only VCC, GND, LED. Result should be backlight on, screen white. If not, your power wiring is the problem.
  3. Add signals one at a time: RESET → CS → A0 → SD → SCK. Until SCK is connected, screen should remain white. The moment SCK is wired, the chip starts initializing. If still white after all 5 are connected, one wire is faulty.
  4. Backlight glow with only GND wired = phantom power leakage. Some signal pin is back-feeding through a protection diode. Disconnect everything, fix VCC, retry.

Wrong size / shifted image

SymptomFix
Only ¼ of screen fills, rest is TV noiseAddress window too big. Set TFT_W=160, TFT_H=128.

"My library doesn't work" — known C3 issue

Both Adafruit_ST7735 and TFT_eSPI can silently re-bind SPI bus pins to their defaults on the ESP32-C3, ignoring custom build flags. If you've configured pins correctly and still get a white screen, switch to raw SPIClass(FSPI) with explicit pin assignments and a hand-rolled init.

This BMO firmware does exactly that — see tftInit() in main.cpp.

The full history of what we tried before this worked
  • Adafruit_ST7735 5-arg software SPI constructor → screen stayed white
  • Adafruit_ST7735 hardware SPI on FSPI → still white. Library re-bound bus pins.
  • TFT_eSPI with USER_SETUP_LOADED + explicit defines → still white
  • 5 different ST7735 tab variants: GREENTAB, BLACKTAB, REDTAB, 18GREENTAB, MINI160x80 → all white
  • Raw bit-banged ST7735 init on our exact pins → first color! But only ¼ of screen because address window was 240×320
  • Switched to hardware SPIClass(FSPI) with our pin map and a manual init → works at 40 MHz

How the animation works

Double-buffered framebuffer

Drawing primitives (fbFillRect, fbFillCircle, etc) go into a 40 KB RGB565 buffer in RAM. The buffer is flushed to the panel in one big SPI burst per frame, so the panel never shows a half-drawn frame.

static uint16_t fb[TFT_W * TFT_H];   // 40 KB

void renderFrame(...) {
  fbFill(C_BG);
  fbDrawEye(...);
  fbDrawSmile(...);
  flushFrame();   // single SPI burst, ~5 ms at 40 MHz
}

Time-driven moods

Each mood is a function from time → face state, not a still pose. The face state struct gets re-evaluated every ~30 ms (≈30 fps), producing real motion like blinking, pupil sway, mouth pulses, and shake.

case MOOD_TALK: {
  s.mouth = FaceState::M_TALK;
  s.mouthOpen = 0.5f + 0.5f * sinf(now * 0.025f);  // mouth oscillates
  s.pupilDy = (int)(1 * sinf(now * 0.01f));        // gentle pupil bob
  break;
}

Auto-cycle script in loop() rotates through these (interrupted by touch):

Audio system new

I2S output pipeline

The C3's I2S peripheral streams 16 kHz mono int16 PCM samples to the MAX98357A. The amp converts I2S to a class-D analog signal that drives the speaker.

void audioInit() {
  i2s_config_t cfg = {
    .mode = I2S_MODE_MASTER | I2S_MODE_TX,
    .sample_rate = 16000,
    .bits_per_sample = I2S_BITS_PER_SAMPLE_16BIT,
    .channel_format = I2S_CHANNEL_FMT_ONLY_LEFT,
    .communication_format = I2S_COMM_FORMAT_STAND_I2S,
    ...
  };
  i2s_pin_config_t pins = {
    .bck_io_num = PIN_I2S_BCLK,   // GP0
    .ws_io_num = PIN_I2S_LRC,     // GP1
    .data_out_num = PIN_I2S_DOUT, // GP2
  };
  i2s_driver_install(I2S_NUM_0, &cfg, 0, NULL);
  i2s_set_pin(I2S_NUM_0, &pins);
}

Volume control

A global g_volume float (0.0..1.0) scales every sample as it's written to I2S. Default is 0.234. To change at runtime:

setVolume(0.5f);   // 50% volume
float v = getVolume();

The variable is volatile and clamped, safe to update from any task or interrupt. A future web UI just calls setVolume(slider/100.0f).

Synthesized toy voices

A library of small "voice" functions in main.cpp gives BMO a bright toy-computer feel with intonation curves on sine tones:

  • playBmoHi() — 3-syllable rising warble that feels like "hi friend!"
  • playBmoWhat() — single rising chirp with question intonation
  • playBmoChant() — 4 quick beeps with a tiny bounce ("BMO BMO BMO BMO")
  • playBmoHooray() — rising 4-note triumph C → E → G → B → high-E sparkle
  • playBmoLaugh() — 5 alternating chirps that descend in pitch
  • playBmoStartled() — sharp "huh!" double-chirp
  • playBmoBeepBoop() — classic two-tone
  • playBmoSigh() — content descending warble
  • playBmoGames() — 9-note playful melody
These synth voices are stand-ins for the tiny generic toy-voice clips baked via the voice clips system. The firmware automatically uses baked clips when available and falls back to synth when missing.

Touch reactions new

Touch classifier

The TTP223 sensor outputs HIGH while touched, LOW when released. The firmware polls this on every frame (~30 Hz) and classifies each touch by hold duration and recent press count:

GestureDetectionReaction
QUICK_POKEHold < 250 ms (single tap)Startled "what?!"
HOLD250–1500 ms holdPetted, content "hi friend!"
LONG_HOLD> 1500 ms holdDeep affection → heart eyes
TICKLE3+ rapid taps within 900 ms windowHelpless laughter

Multi-phase reactions

Each gesture triggers a sequence of timed phases that tell a small story, applying classic animation principles:

  • Anticipation — a tiny pre-motion before the main reaction (squash before recoil)
  • Squash & stretch — body deforms slightly under motion
  • Follow-through — body keeps wobbling slightly after eyes settle
  • Secondary motion — cheeks, blush, eyelids react with offset timing
  • Eye tracking — pupils dart toward the touch direction, then track to the user
  • Particles — sparse hearts, stars, and blush dots support the reactions without covering the face

Quick poke reaction breakdown

  1. Squash 80 ms — body compresses down, mouth flattens (anticipation)
  2. Recoil 220 ms — launches upward with eased motion and a clean gasp face
  3. Dart 600 ms — pupils snap left → right → center looking for the culprit, body wobbles
  4. Recover 700 ms — double-blink, soft smile, blush blooms, gentle breathing

Hold reaction breakdown

  1. Glance 220 ms — pupils track toward touch, eyes start to squint happy, content sigh tone
  2. Relax 800 ms — pupils return forward, eyes close to crescents, mouth opens gently, "hi friend!" plays
  3. Purr 1500 ms — full breathing animation, pink dot puffs trickle up off the cheeks

Long hold reaction breakdown

  1. Pet 1000 ms — same warm-up as hold, "BMO BMO BMO BMO" chant plays
  2. Swell 700 ms — eyes start opening wider, body slowly rises, "Hooray!" rising scale
  3. Sting 250 ms — sparkle high note + tiny heart pop, "Who wants to play video games?" melody
  4. Bashful hold 1700 ms — brief heart eyes settle into shy crescent eyes, sparse hearts, soft bob

Tickle reaction breakdown

  1. Burst 350 ms — sharp "ha!" with star confetti exploding from mouth
  2. Sustained 1900 ms — bouncing laughter, stars sprinkle on each peak, "haha" beeps trigger when wave crests
  3. Exhausted slump 300 ms — body sags, mouth closes to a tired smile

Particle system

Up to 12 simultaneous particles fly outward with simple physics. Each has position, velocity, lifetime, color, and shape kind:

particlesEmit(6, MOUTH_CX, MOUTH_Y, /* kind */ 1, C_HEART);
// kind 0 = star, 1 = heart, 2 = exclamation, 3 = soft dot

Particles step each frame: position += velocity, velocity.y += gravity, life -= dt. They fade out and shrink as life approaches 0. Used in all four touch reactions.

Chained reactions

The reaction loop polls touch on every frame, so if you start a new gesture mid-reaction (e.g. tickle while a hold is purring), BMO breaks out and starts the new reaction immediately. This makes BMO feel responsive instead of "deaf" during animations.

Voice clips new

How it works

You can generate a generic high toy-computer voice pack with tools/generate_tiny_voice.py, then run tools/bake_audio.py. The baker converts clips to 16 kHz 4-bit IMA ADPCM and writes src/audio_clips.h. The firmware decodes ADPCM on the fly and falls back to procedural synth if a clip is missing.

File naming

The filename (minus extension) determines what gesture plays the clip:

bmo_hooray.wavLONG_HOLD middle"Hooray!" or "Yay!"
bmo_games.wavLONG_HOLD end"Who wants to play video games?"
bmo_laugh.wavTICKLEGiggling / "hee hee hee"

One-time setup

brew install ffmpeg     # used by the generator and by MP3/WAV conversion

Bake workflow

cd firmware/bmo_face_anim

# 1. Drop WAV files into ./audio/
cp ~/Downloads/bmo_hi_friend.wav audio/
cp ~/Downloads/bmo_what.wav audio/

# Or generate the generic tiny toy-voice pack
python3 tools/generate_tiny_voice.py

# 2. Bake them into a compressed C header
python3 tools/bake_audio.py

# 3. Reflash
pio run -t upload

Output footprint

Each second of clip is about 8 KB of flash with 4-bit ADPCM. The current generated seven-clip phrase pack is about 16 KB total, which is intentionally tiny for the ESP32-C3.

Quality notes

  • The ADPCM format is small-speaker quality: clear enough for short phrases, intentionally not hi-fi.
  • The generator makes a BMO-inspired generic toy voice, not an exact clone of any actor or copyrighted character voice.
  • If a clip sounds quiet/dull after baking, your source WAV is probably already too quiet. Use a louder source or regenerate.
  • Stereo files are auto-converted to mono. Sample rates auto-resample to 16 kHz.
Avoid redistributing copyrighted character voice samples. The included generator is intentionally generic and toy-like.

Extending BMO

Add a new mood

  1. Add an enum entry to Mood in main.cpp
  2. Add a name in moodName()
  3. Add a case in playMood() that sets FaceState fields as a function of now or t
  4. Optionally add a jingle to the jingleFor() dispatcher
  5. Add an entry to the script[] array in loop() with a duration

Add a new touch gesture

  1. Add a TouchKind enum entry
  2. Update pollTouch()'s classification logic (e.g. detect double-tap, force-press if you have analog input)
  3. Add a new case in playReaction() with phase animations
  4. Optionally add a clip name to playClipOrSynth() calls

Add a custom voice clip

  1. Drop the WAV into audio/your_clip_name.wav
  2. Run python3 tools/bake_audio.py
  3. In main.cpp, call it with playClip(findClip("your_clip_name"))

Hook into web simulator

The repo includes a Next.js + Tailwind simulator under web/ that connects via Web Serial. The firmware can emit JSON-over-USB lines for each rendered mood and accept inbound mood/sensor commands. To enable simulator mode, swap the body of renderFrame() — same face state, different output backend.

Add WiFi + cloud STT/LLM/TTS (planned)

The conversation pipeline is sketched out:

  1. INMP441 mic captures audio while touch is held → 3 sec int16 PCM in RAM
  2. HTTPS POST to api.openai.com/v1/audio/transcriptions (Whisper)
  3. POST transcript to api.anthropic.com/v1/messages (Claude)
  4. POST reply to api.openai.com/v1/audio/speech (TTS)
  5. Stream resulting audio bytes directly to I2S → speaker

Memory budget on the C3 is tight (~340 KB needed, 320 KB available) — workaround is to record at 8 kHz or stream uploads instead of buffering.

Move to a different screen

Update TFT_W, TFT_H, X_OFFSET, Y_OFFSET, and the MADCTL byte. The 40 KB framebuffer scales linearly — 240×240 round display = 115 KB, still fits the C3's RAM.

Move to ESP32-S3 with camera

The S3 has 8 MB PSRAM and a DVP camera interface. Port involves changing 5 pin numbers (SPI bus differs), adding camera init, and routing camera frames to the screen via a "look at me" mode. Same face firmware works.
